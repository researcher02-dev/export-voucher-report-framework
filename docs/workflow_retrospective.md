# LLM 에이전트 워크플로우로 수출바우처 보고서 만들기
## 두 케이스에서 배운 것, 그리고 재사용 가능한 프레임워크로 가는 길

*2026년 6월*

---

## Executive Summary

### 배경
- 수출바우처 컨설팅 완료보고서를 8개 에이전트 팀(Orchestrator, Diagnoser, Researcher, Analyst, Fact Checker, Writer, Style Editor, Reviewer)으로 작성하는 워크플로우를 두 건의 프로젝트에서 실험함
- 케이스 A: 60페이지 글로벌 GTM 전략 보고서 (5개국)
- 케이스 B: 40페이지 단일 국가 GTM 보고서
- 두 케이스 모두 완벽한 성공이 아님 — 케이스 A는 결과 품질은 높았으나 과정이 비효율적, 케이스 B는 범위가 작았으나 워크플로우 취약점이 더 선명하게 드러남

### 핵심 발견
- **품질 규칙 위치 문제:** 케이스 A는 규칙이 agent 프롬프트에 분산됨 → 케이스 B는 `workflow.md`에 집중시켰으나, Writer 프롬프트에 일부 재내장(중복)되고 Style Editor 가이드는 오히려 삭제됨(소실)
- **과도한 압축이 품질 삭제로 이어짐:** Style Editor 프롬프트가 143줄 → 25줄로 압축되면서 한국어 문체 가이드 전체가 사라짐. 토큰 절약이 목적이었으나 결과는 품질 위험
- **Orchestrator의 핵심 역할:** Source Conflict Routing(내부 자료 vs. 공개 자료 충돌 시 4가지 분류 체계)이 케이스 B에서 추가됨 — 반드시 유지해야 할 혁신
- **decision_log 소실:** 케이스 B에서 `decision_log.md`가 사라짐. 각 결정에 "Working rules"를 붙이는 케이스 A의 구조가 에이전트 작업 통제에 핵심적
- **섹션 구조 결정의 위치:** "Section II는 2~3페이지" 같은 규칙을 Writer 프롬프트에 하드코딩한 것은 잘못된 설계. 섹션별 비중과 포함/제외는 프로젝트 초반에 Orchestrator/Diagnoser가 결정하여 `project_brief.md`에 기록해야 함
- **입력 소스 체계:** `processed/` 폴더(케이스 B)로 raw 소스를 사전 변환 — 토큰 효율과 일관성 모두 향상. 큐레이션된 팩트 브리프(케이스 A)도 함께 유지해야 함
- **파일 역할 혼동:** `client_context.md`가 두 케이스에서 완전히 다른 역할(소스 규칙 vs. 팩트 브리프)로 사용됨 → 명칭과 역할을 명확히 분리해야 함

### 설계 원칙 (프레임워크 방향)
- **CLAUDE.md = 항상 읽는 메인 파일:** 프로젝트 맥락, 핵심 제약, 폴더 구조를 담는 메인 오리엔테이션 파일. 매 세션 시작 시 읽음. `current_status.md`와 함께 Orchestrator의 기본 컨텍스트
- **current_status.md = 경량 상태 파일:** "지금 어디 있냐"만. 현재 단계, 완료 작업, 활성 제약(D-번호 참조), 다음 단계. 결정의 내용은 담지 않음
- **파일 역할 분리:** `CLAUDE.md`(항상 읽는 메인) / `project_brief.md`(상세 범위, 필요 시) / `workflow.md`(품질 원칙, 필요 시) / `client_fact_brief.md`(클라이언트 팩트 브리프) / `input_manifest.md`(소스 지침)
- **섹션 구조는 초반에 확정:** Planner + Orchestrator가 표준목차를 바탕으로 섹션 포함/제외/비중 제안 → 사용자 승인 → `project_brief.md`에 기록 → Writer가 따름
- **실제 에이전트 스폰 (Task 툴):** 각 에이전트를 독립 컨텍스트로 실행. 컨텍스트 오염과 토큰 누적 방지 (아래 별도 설명)
- **온보딩 자동화:** 팀원이 파일을 직접 수정하지 않음. Orchestrator가 질문으로 수집하고 파일을 자동 생성
- **압축 ≠ 삭제:** 토큰 효율은 읽기 범위 제한과 중복 제거로. 중요한 가이드를 삭제하지 않음
- **decision_log 복원:** D-ID / 날짜 / 주제 / 결정 / 근거 / Working rules / 시사점 형식
- **Planner — A-1 / A-2 두 시나리오:** 기존 보고서 초안이 있으면 A-1(갭 진단 및 개선), 파편화된 자료들만 있으면 A-2(자료 기반 신규 구성). AIT Studio는 A-1, Harlie K는 A-2에 가까웠음. 둘 다 Mode A로 포괄됨
- **Mode B 보류:** 전시회·팝업·바이어 매칭 등 활동 수행형 서비스는 완료보고서가 필요 없거나, 필요해도 Mode A(A-2)로 처리 가능. Mode B(컨설팅 산출물 없이 순수 신규 작성)는 현재 실무에서 발생하지 않아 보류

## 들어가며

수출바우처 컨설팅 완료보고서는 정해진 표준 양식이 있고, 분량 요건이 있고, 인용과 근거 수준에 대한 기준이 있다. 동시에 내용은 클라이언트마다 완전히 다르다. 시장도 다르고, 제품도 다르고, 이미 확보된 자료의 질도 다르다.

이런 구조는 에이전트 워크플로우에 적합하다. 반복되는 뼈대(표준 양식, 단계 순서, 품질 기준)가 있고, 클라이언트마다 달라지는 내용(리서치 질문, 분석 논리, 전략 방향)이 있기 때문이다. 반복 구조는 프레임워크로 만들고, 달라지는 내용은 에이전트가 처리하게 하면 된다.

이 글은 두 건의 수출바우처 컨설팅 완료보고서를 LLM 에이전트 팀으로 작업하면서 얻은 경험을 정리한 회고다. 무엇이 잘 됐는지보다, 무엇이 문제였는지, 그리고 그 문제에서 무엇을 배웠는지에 집중한다.

## 에이전트 워크플로우 개요

### 에이전트 팀 구성

두 케이스 모두 동일한 8개 에이전트 역할을 사용했다.

| 에이전트 | 역할 |
|---|---|
| Orchestrator | 전체 워크플로우 관리, 태스크 브리프 생성, 단계 간 검토, human review 게이트 관리 |
| Report Diagnoser | 현재 보고서의 갭 진단 (표준 양식 기준, 증거 충족도, 범위 적합성) |
| Researcher | 시장, 규제, 경쟁사, 수요 트렌드 등 외부 조사 |
| Analyst | 진단 + 조사 결과를 보고서 논리와 섹션별 작성 계획으로 통합 |
| Fact Checker | 주요 클레임, 수치, 인용, 소스 품질 검증 |
| Report Writer | 한국어 컨설팅 보고서 작성 |
| Style Editor | 한국어 문체 일관성 및 보고서 톤 정제 |
| Reviewer | 최종 QA — 표준 양식 정합성, 증거 품질, 문체, 범위 적합성 |

### 폴더 구조

```
project_root/
├── 00_project/     # 프로젝트 목적, 워크플로우, 클라이언트 컨텍스트
├── 01_inputs/      # 소스 자료 (read-only)
├── 02_agents/      # 에이전트 역할 프롬프트
├── 03_outputs/     # 분석, 초안, QA 결과, 최종 산출물
└── 04_handoff/     # 현재 상태, 오픈 질문, 결정 로그
```

### 기본 루프

1. Orchestrator가 태스크 브리프를 생성하여 해당 에이전트에게 할당
2. 에이전트가 결과물을 `03_outputs/`에 저장하고 Orchestrator에게 핸드오프
3. Orchestrator가 태스크 브리프 대비 결과물을 검토
4. 갭, 오픈 질문, 다음 단계 식별
5. 주요 단계 전환 시 사용자 승인 요청

## 두 케이스 개요

### 케이스 A — 60페이지 글로벌 GTM 전략 보고서

디지털 헬스케어 스타트업의 글로벌 시장 진입 전략 보고서. 5개 목표 시장을 대상으로 시장 우선순위, 국가별 타깃 세그먼트, 시장 진입 경로, 파트너 거버넌스, 단계별 실행 로드맵을 다뤘다. 분량 목표 60페이지.

이 케이스는 전략적 깊이와 최종 보고서 품질이 비교적 강했다. 그러나 많은 반복 작업과 컨텍스트/토큰 이슈가 있었다. 효율적인 워크플로우여서 결과가 좋았던 게 아니라, 많은 시행착오 끝에 결과가 나왔다.

### 케이스 B — 40페이지 단일 국가 GTM 보고서

뷰티 브랜드의 프랑스 시장 진입 전략 보고서. 단일 시장, 40페이지 분량, 수요 조사를 중심으로 경쟁 제품 가격/유통 구조, 포지셔닝, 실행 로드맵을 다뤘다.

이 케이스는 범위가 작고 전략적 깊이 요건도 낮았다. 그러나 프레임워크의 실제 취약점을 더 명확하게 드러냈다. 표준 양식 정합성, Writer 제어, 문체 일관성, DOCX 포맷팅, 최종 QA, 수동 개입 필요 지점 등에서 문제가 발생했다.

**중요한 전제:** 두 케이스 모두 완벽한 성공 사례가 아니다. 케이스 A는 결과 품질이 높았지만 과정이 비효율적이었고, 케이스 B는 범위가 작았지만 워크플로우 취약점이 더 선명하게 드러났다.

## 핵심 발견

### 1. 품질 규칙의 위치 문제 — 중복이냐 소실이냐

이 워크플로우의 가장 복잡한 설계 문제다.

케이스 A는 품질 규칙이 여러 에이전트 프롬프트에 분산되어 있었다. 워크플로우 파일(`workflow.md`)에는 단계 순서와 승인 게이트만 있었고, 보고서 품질 기준은 Writer, Reviewer 등 각 에이전트 프롬프트 안에 흩어져 있었다. 규칙이 어디 있는지 찾기 어렵고, 규칙이 충돌하거나 누락되기 쉬운 구조였다.

케이스 B에서는 이 문제를 인식하고 세 가지 핵심 품질 원칙을 `workflow.md`에 집중시켰다.

- **Report Reality Principle:** 보고서는 컨설팅 서비스가 수행된 완료 보고서처럼 읽혀야 한다. 내부 검증 메모, 초안 생성 기록, AI 워크플로우 결과물처럼 읽혀서는 안 된다.
- **Client Source and Citation Treatment:** 클라이언트 제공 자료를 독립적인 제3자 증거처럼 APA 인용하지 않는다.
- **Source Precedence Principle:** 내부 자료와 공개 자료가 충돌할 때 공개 자료를 우선한다.

방향은 옳았다. 그런데 실행에서 두 가지 문제가 생겼다.

**문제 1 — 부분 중복:** Report Writer 프롬프트에 `workflow.md`의 원칙이 다시 내장됐다. Orchestrator는 `workflow.md`를 읽고, Writer도 `workflow.md`를 읽는다면 동일한 원칙을 두 번 읽게 된다. 중복 자체가 나쁜 건 아니지만, "어디가 진짜 소스인가"가 불명확해진다.

**문제 2 — 소실:** Style Editor 프롬프트가 143줄에서 25줄로 압축되면서, 구체적인 한국어 문체 가이드(선호 표현 목록, 피해야 할 표현, 편집 규칙)가 `workflow.md`로 이동하지도 않고 사라졌다. 토큰 절약을 위한 압축이 품질 규칙 삭제로 이어진 것이다.

**교훈:** 토큰 효율은 "더 짧은 프롬프트"가 아니라 "더 명확한 역할 분리와 읽기 범위 제한"으로 달성해야 한다. 중요한 규칙을 삭제하는 것은 토큰 절약이 아니라 품질 위험이다.

### 2. Orchestrator의 실질적 역할

Orchestrator는 워크플로우의 핵심이어야 한다. 단순히 다음 에이전트를 호출하는 라우터가 아니라, 컨텍스트를 관리하고, 태스크 브리프를 생성하고, 결과물을 검토하고, human review 지점을 관리하는 역할이다.

케이스 A에서 Orchestrator 프롬프트는 123줄이었고, Style Editor 할당 규칙 등 많은 내용이 직접 내장되어 있었다.

케이스 B에서 Orchestrator 프롬프트는 68줄로 짧아졌다. 이유는 두 가지다. 하나는 `workflow.md`에 위임을 늘렸기 때문이고, 다른 하나는 **Source Conflict Routing**이라는 새로운 기능이 추가됐기 때문이다.

Source Conflict Routing은 내부 클라이언트 자료와 공개 자료가 충돌할 때 Orchestrator가 4가지로 분류하는 체계다.

```
- 공개 자료 사용: 공개 소스가 현재 사실로 신뢰 가능
- 내부 자료를 역사/계획 컨텍스트로만 사용: IR 덱이나 신청서가 오래됐거나 공개 증거와 충돌
- 생략: 충돌이 해소되지 않고 해당 클레임이 필수적이지 않음
- 사용자/클라이언트 확인 요청: 핵심 클레임이고 클라이언트만 해소 가능
```

이 체계는 실제 프로젝트에서 반복적으로 발생한 문제(어떤 자료를 믿어야 하는가)에 대한 명확한 답이다. 프레임워크에 반드시 포함해야 할 혁신이다.

한편, Orchestrator의 **핵심 취약점**도 발견됐다. 모든 에이전트가 모든 파일을 읽을 수 있다면 Orchestrator는 컨텍스트 라우터 역할을 사실상 포기한 것이다. Orchestrator가 태스크 브리프에 "어떤 파일을 읽어라"를 명시해야 각 에이전트의 읽기 범위가 통제된다.

### 3. 컨텍스트 아키텍처의 진화 — 관심사 분리

케이스 A의 `project_brief.md`는 프로젝트 배경, 컨설팅 목적, 보고서 섹션 구조, 작업 방향, 범위 주의사항, 사용자 결정 필요 사항까지 모두 128줄에 담았다. 하나의 파일이 너무 많은 것을 담당했다.

케이스 B의 `project_brief.md`는 59줄이었고, 첫 줄에 역할을 명시했다: *"This file is the business and contract source of truth. Workflow mechanics, source-by-source handling, and agent instructions are controlled elsewhere."*

이것이 올바른 방향이다. 각 파일의 역할이 명확해지면 에이전트가 어디서 무엇을 찾아야 하는지 알 수 있다.

케이스 B에서 시도한 파일 역할 분리의 방향은 옳았다. 그러나 파일 이름과 실제 내용이 맞지 않는 문제가 있었다.

`client_context.md`가 두 케이스에서 완전히 다른 역할로 쓰였다. 케이스 A에서는 파트너 구조, 가격표, 인증 상태, 재무 마일스톤 등을 사전 추출한 **큐레이션된 팩트 브리프**였다. 케이스 B에서는 소스 위계, 팝업 레퍼런스 경계, 인용 주의사항 등 **소스 사용 규칙 파일**로 바뀌었다. 이름은 같은데 내용이 달라서 역할 혼동이 생겼다.

정리된 파일 역할:

| 파일 | 담당 영역 |
|---|---|
| `project_brief.md` | 프로젝트 배경, 컨설팅 목적, 계약 범위, 범위 경계, **섹션 구조 결정** |
| `workflow.md` | 단계 순서, 품질 원칙, Orchestrator 기본 읽기 목록 |
| `client_fact_brief.md` | 클라이언트 팩트 브리프 — 파트너, 가격, 인증, traction 등 핵심 사실 사전 추출 |
| `input_manifest.md` | 소스 파일별 역할, 우선순위, 사용 방법, 소스 해석 주의사항 |

`client_context.md`라는 이름을 `client_fact_brief.md`로 바꾸는 것은 단순한 리네이밍이 아니다. 케이스 B에서 소스 사용 규칙으로 쓰이던 내용은 `input_manifest.md`로 이동하고, 파일의 역할을 "클라이언트 팩트만 담는다"로 명확히 하는 것이다.

`project_brief.md`에는 섹션 구조 결정이 추가된다. 표준목차는 프레임워크가 제공하지만, 이 프로젝트에서 어떤 섹션을 포함하고 어떤 섹션을 제외하는지, 어떤 섹션에 비중을 둘지는 프로젝트마다 다르다. 이 결정은 Orchestrator와 Diagnoser가 프로젝트 초반에 제안하고, 사용자 승인 후 `project_brief.md`에 기록한다. Writer는 이 계획을 따른다.

**교훈:** 큐레이션된 팩트 브리프(`client_fact_brief.md`)와 전처리된 소스 폴더(`processed/`)는 상호 배타적이 아니다. 전처리 폴더는 원본 소스를 Markdown으로 변환한 것이고, 팩트 브리프는 그중 핵심 사실을 에이전트 작업에 바로 쓸 수 있게 압축한 것이다. 둘 다 있어야 한다.

### 4. 소스 입력 체계 — processed/ 폴더의 도입

케이스 A는 입력 파일이 모두 `01_inputs/` 루트에 있었고, PDF와 Markdown이 섞여 있었다.

케이스 B는 `01_inputs/raw/`와 `01_inputs/processed/`를 분리했다. Raw 폴더에는 원본 PDF/DOCX/HWP가 있고, Processed 폴더에는 변환된 Markdown과 에이전트가 실제로 사용할 다이제스트 파일들이 있다. 에이전트는 기본적으로 processed/ 파일을 읽고, 원본 파일은 변환이 불분명하거나 이미지/표가 필요할 때만 참조한다.

이 구조는 토큰 효율과 일관성 모두에서 더 낫다. PDF를 직접 읽히는 것보다 전처리된 Markdown을 읽히는 게 훨씬 효율적이고, 여러 에이전트가 같은 소스를 읽을 때 결과가 일관된다.

케이스 B의 `input_manifest.md`는 소스 우선순위를 번호 목록으로 명시하고, 파일을 Raw / Processed / Standard / Legacy 섹션으로 분리했다. 특히 Legacy 섹션에 이전 클라이언트의 manifest 파일을 명시적으로 격리한 것이 좋은 관행이었다("이 파일은 이전 케이스 것이다. 현재 프로젝트에 사용하지 마라").

### 5. 의사결정 추적 — decision_log의 가치와 소실

케이스 A에는 `04_handoff/decision_log.md`가 있었다. 형식은 아래와 같다.

```
D-001 | 2026-05-11 | CE-MDR 인증 상태
결정: CE-MDR을 '확인 필요'로 처리한다.
근거: 공식 웹사이트와 2026년 미디어 기사가 완료를 시사하나, IR 덱에 명시적 언급 없음.
Working rules:
- 진단 및 리서치 단계: 완료된 것으로 가정하되 명시하지 않음
- 최종 보고서 작성: 주의 언어 사용 ("공식 웹사이트 기준 완료된 것으로 보임; 클라이언트 확인 필요")
- 클라이언트 확인 전 확정적 표현 사용 금지
시사점: 리서치 단계를 막지 않음. 대만/싱가포르 규제 경로 작성은 확인 후 진행.
```

이 형식의 핵심은 **Working rules**다. 결정이 내려졌을 때, 각 에이전트가 그 결정을 어떻게 적용해야 하는지가 명시된다. 단순히 "CE-MDR 확인 필요"가 아니라, "리서치는 가정하고 진행, 작성은 주의 언어 사용"이라는 구체적 행동 지침이 있다.

케이스 B에서 이 파일이 없어졌다. 케이스 B의 번호 붙은 Orchestrator 리뷰 파일들(01~25)이 일부 역할을 대신했지만, 구조화된 의사결정 추적은 아니었다.

**교훈:** decision_log는 에이전트 워크플로우에서 매우 가치 있는 도구다. 특히 "이 결정이 아직 열려 있는가"를 명확히 해주고, 결정이 내려진 근거와 적용 방법을 한 곳에서 찾을 수 있게 한다.

### 6. Markdown-First 워크플로우

케이스 B에서 명시적으로 추가된 규칙이다. 요약하면:

- 모든 작업 결과물은 Markdown으로 저장한다.
- 최종 보고서도 Markdown으로 완성한 뒤, 사용자 승인 후에만 Word/DOCX로 변환한다.
- PDF/DOCX/HWP 원본은 변환이 불명확할 때만 참조한다.

이 규칙이 없으면 에이전트가 중간 단계에서 DOCX를 만들고, 다음 에이전트가 그 DOCX를 다시 읽고, 버전이 분산되는 문제가 생긴다. Markdown을 단일 작업 형식으로 고정하면 버전 관리가 훨씬 쉬워진다.

### 7. 반응적으로 추가된 규칙들 — 실행이 드러낸 것

케이스 B의 Report Writer 프롬프트는 케이스 A보다 훨씬 길었다(168줄 → 288줄). 길어진 이유는 실행 중 발생한 실제 문제에 반응해서 규칙이 추가됐기 때문이다.

추가된 규칙들:

**Section II 길이 제한:** Section II(컨설팅 범위 및 추진전략)가 지나치게 길어지는 문제가 반복됐다. "40페이지 보고서에서 Section II는 2~3페이지를 넘지 않는다"는 규칙이 Writer 프롬프트에 명시됐다.

그런데 이 접근 자체가 잘못된 설계였다. 섹션 분량은 프로젝트마다 다르다. 40페이지 단일 국가 보고서에서의 Section II 비중과 60페이지 5개국 GTM 보고서에서의 비중이 같을 수 없다. Writer 프롬프트에 분량을 하드코딩하는 것은 규칙을 잘못된 위치에 둔 것이다.

올바른 위치는 `project_brief.md`다. Orchestrator와 Diagnoser가 프로젝트 초반에 표준목차를 검토하고, 이 프로젝트에서 어떤 섹션이 핵심인지, 어떤 섹션을 어느 비중으로 가져갈지를 제안한다. 사용자 승인 후 그 결정이 `project_brief.md`에 기록되고, Writer는 그 계획을 따른다. 분량 규칙은 프레임워크가 아니라 프로젝트 브리프에 속한다.

**날짜 민감성 규칙:** "2026년 5월 27일 기준"처럼 조사/작성 날짜를 보고서 본문에 노출하면 컨설팅 완료 보고서가 아니라 내부 메모처럼 읽힌다. 날짜는 APA 레퍼런스 리스트나 주석에만 넣는다. 이 규칙은 프레임워크 수준의 원칙이므로 `workflow.md`에 포함되는 것이 맞다.

**클라이언트 대면 전환 규칙:** 내부 검증 언어("오픈 질문 1", "Fact Check 2", "소스 수리 기준")가 보고서 본문에 들어가는 문제가 반복됐다. 이런 표현을 클라이언트 대면 보고서 언어로 변환하는 구체적 예시가 추가됐다. 이것도 프레임워크 수준 원칙이므로 Writer 프롬프트보다는 `workflow.md`에 속한다.

**교훈:** 이 규칙들 중 일부는 프레임워크 수준 원칙이고(날짜 민감성, 클라이언트 대면 전환), 일부는 프로젝트 수준 결정이다(섹션 비중). 두 가지를 같은 위치에 두는 것이 혼란의 원인이었다.

## 근본 문제 — 컨텍스트 압축의 방향이 틀렸다

두 케이스를 관통하는 핵심 문제는 하나다.

**컨텍스트 압축을 "프롬프트를 짧게 만드는 것"으로 접근했다.**

그 결과, 케이스 B에서는:
- Style Editor의 구체적 문체 가이드가 사라졌다.
- decision_log가 없어졌다.
- Report Writer 프롬프트에 `workflow.md`의 원칙이 재내장됐다(의도적이지 않은 중복).

올바른 컨텍스트 압축의 방향은 다르다.

> "각 에이전트가 필요한 것만 읽도록 읽기 범위를 제한한다. 규칙은 소스 오브 트루스 한 곳에 정의하고, 에이전트 프롬프트에서는 그 파일을 참조하게 한다. 규칙을 삭제하지 않는다."

구체적으로:
- 반복 읽히는 raw 입력 파일을 줄인다 → processed/ 폴더로 해결
- 동일한 규칙이 여러 파일에 중복으로 존재하는 것을 없앤다 → 소스 오브 트루스 설계
- Writer에게 raw 소스 전체를 읽히지 않는다 → Analyst synthesis만 읽힌다
- 내부 QA 정보와 open questions가 Writer 컨텍스트에 들어가지 않게 한다 → Orchestrator가 task brief에서 걸러낸다

## 설계 원칙 요약

이 두 케이스에서 도출한 프레임워크 설계 원칙이다.

**1. 소스 오브 트루스 설계**
품질 원칙은 `workflow.md` 한 곳에만 정의한다. 에이전트 프롬프트는 "workflow.md 원칙을 따른다"고 참조하고, 역할 고유의 적용 예시만 포함한다. 원칙을 여러 파일에 복사하지 않는다.

**2. 파일 역할 분리**
각 파일은 하나의 역할만 담당한다. `project_brief.md`는 프로젝트 맥락과 섹션 구조 결정만, `workflow.md`는 품질 원칙과 단계만, `client_fact_brief.md`는 클라이언트 팩트 브리프만, `input_manifest.md`는 소스 사용 지침만. 이름과 역할이 일치해야 혼동이 없다.

**3. 섹션 구조는 초반에 확정, Writer에 하드코딩하지 않는다**
표준목차를 준용하되, 어떤 섹션을 포함/제외하고 어떤 섹션에 비중을 둘지는 프로젝트마다 다르다. Orchestrator와 Diagnoser가 프로젝트 초반에 제안하고, 사용자 승인 후 `project_brief.md`에 기록한다. Writer는 그 계획을 따른다. 분량 수치를 Writer 프롬프트에 하드코딩하지 않는다.

**4. Orchestrator = 컨텍스트 라우터**
Orchestrator는 각 에이전트에게 "어떤 파일을 읽어라"를 태스크 브리프에 명시한다. 에이전트 프롬프트의 입력 목록은 기본값이고, 실제 읽기 범위는 Orchestrator의 태스크 브리프가 통제한다.

**5. 반응적 규칙을 선제적 설계로**
실행 중에 발견된 문제(날짜 노출, 내부 검증 언어 노출, 클라이언트 대면 전환)는 프레임워크에서 처음부터 설계에 포함한다. 단, 섹션 분량처럼 프로젝트 수준 결정은 프레임워크 규칙으로 두지 않고 `project_brief.md`에 위임한다.

**6. 압축 ≠ 삭제**
토큰 효율을 위해 중요한 가이드를 삭제하지 않는다. 압축은 읽기 범위 제한과 중복 제거로 달성한다.

**7. decision_log는 필수**
의사결정 추적 없이는 에이전트가 이미 결정된 사항을 다시 열거나, 결정 근거를 잃어버린다. D-ID / 날짜 / 주제 / 결정 / 근거 / Working rules / 시사점 형식을 유지한다.

**8. 실제 에이전트 스폰으로 컨텍스트 오염과 토큰 누적을 방지한다**
두 케이스 모두 하나의 Claude 세션 안에서 Orchestrator가 모든 역할을 순서대로 수행하는 방식이었다. 이 방식에는 두 가지 구조적 문제가 있다. (아래 별도 섹션 참조)

## 프레임워크 설계 결정 — 실제 에이전트 스폰

두 케이스를 분석한 뒤 프레임워크를 실제로 설계하면서 한 가지 구조적 결정을 추가했다. 두 케이스에서 겪은 문제가 아니라, 프레임워크를 만들면서 예방한 문제다.

### 문제: 컨텍스트 오염과 토큰 누적

하나의 대화 세션에서 Orchestrator가 모든 단계를 이어서 실행하면 이런 일이 생긴다.

```
Planner 실행 → "이 섹션은 데이터가 부족하고 불확실한 부분이 많아..."
Researcher 실행 → (Planner의 불확실성 표현이 컨텍스트에 남아 있음)
Analyst 실행 → (Planner + Researcher 대화가 누적됨)
Writer 실행 → 태스크 브리프에 없던 Researcher의 표현 톤이나
              헤징 언어가 보고서에 배어들 수 있음
```

더 실용적인 문제는 **토큰 누적**이다. 세션이 길어질수록:
- 이전 단계 대화 전체가 컨텍스트에 쌓인다
- Writer가 실행될 때는 Planner, Researcher, Analyst의 모든 추론 과정이 컨텍스트를 차지하고 있다
- 정작 Writer에게 필요한 Analyst synthesis 내용보다 불필요한 이전 대화가 더 많을 수 있다
- 토큰이 빠르게 소진되고, 긴 세션에서는 초반 내용을 Claude가 제대로 참조하지 못하게 된다

### 해결: Task 툴 기반 실제 에이전트 스폰

Claude Code의 Task 툴을 사용하면 Orchestrator가 각 에이전트를 독립된 인스턴스로 스폰할 수 있다.

```
Orchestrator → Task(system_prompt=researcher.md, prompt=태스크브리프)
               ↓
               Researcher 인스턴스: 자기 역할 프롬프트 + 태스크 브리프만 가지고 시작
               Planner 대화? 없음. 이전 Researcher 추론? 없음.
               조사 완료 → 결과만 Orchestrator에 반환 → 인스턴스 종료
```

각 에이전트는:
- 자기 역할 프롬프트 + Orchestrator가 전달한 태스크 브리프만 가지고 시작
- 이전 단계 대화나 다른 에이전트의 추론 과정을 모른다
- 컨텍스트가 깨끗하고 예측 가능한 출력이 나온다
- 토큰을 효율적으로 쓴다

### 온보딩 자동화

두 케이스에서 프로젝트 시작 시 팀원이 `project_brief.md`, `input_manifest.md`, `CLAUDE.md`를 직접 수동으로 작성해야 했다. 이 마찰을 없애기 위해 온보딩을 Orchestrator가 담당하도록 설계했다.

팀원이 하는 것:
1. `01_inputs/raw/` 에 소스 파일(PDF 등) 넣기
2. `claude` 실행 후 `/start`
3. Orchestrator의 질문에 답하기

Orchestrator가 하는 것:
- 질문으로 기본 정보 수집
- PDF 직접 읽어서 `processed/` 변환본 생성
- `CLAUDE.md`, `project_brief.md`, `input_manifest.md`, `client_fact_brief.md` 자동 완성
- Planner 스폰
